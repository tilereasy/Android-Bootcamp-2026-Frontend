# Техническое задание на разработку программного продукта для планирования встреч
Команда: t34M4 n3 R4k0V
## 1. Общие описание
Необходимо разработать программный продукт для планирования встреч сотрудников внутри компании. Система состоит из клиентского мобильного приложения и серверной части.

## 2. Клиентская часть (Android-приложение)
Клиентом является **Android-приложение**, которое должно быть реализовано на **Kotlin** или **Java** с использованием **адаптивной верстки**.

Приложение должно предоставлять интуитивно понятный интерфейс, в котором реализован следующий функционал:

*   **Регистрация и авторизация** сотрудника.
*   **Настройка личного профиля**.
*   **Создание встречи** с возможностью приглашения других сотрудников. Приглашенные сотрудники должны иметь возможность дать бинарный ответ (**принял** / **не принял**).
*   **Просмотр списка активных приглашений** на встречи с возможностью дать ответ о возможности посещения.
*   **Просмотр собственного расписания встреч** в разрезе дня, недели и месяца.

**Важное условие:** временные слоты для встреч разбиты строго по часам. Например, можно создать встречу на 9:00–10:00, но создание встречи на 9:10, 9:05 и т.д. — невозможно.

## 3. Серверная часть
Сервер представляет собой **микросервис на Spring Boot**, работающий с СУБД **PostgreSQL** или **H2**.

Серверное приложение должно:
*   Предоставлять API, необходимый для полноценной работы мобильного приложения.
*   Осуществлять все основные операции с данными в базе через **Spring Data JPA**.
*   Использовать **Liquibase** для создания схемы базы данных и ее предзаполнения.

# User Stories → Use Cases → Dev Tasks (Android + Backend)

## Роли
- **Сотрудник** (основной пользователь приложения)
- **Система** (мобильное приложение + сервер)

---

## 1) Регистрация

### User Story
Как сотрудник, я хочу зарегистрироваться, чтобы получить доступ к системе планирования встреч.

### Use Cases
**UC-1.1 Регистрация сотрудника**
- Предусловия: сотрудник не авторизован.
- Основной сценарий: открыть регистрацию → ввести данные → отправить → получить успешный ответ → попасть на экран входа/в приложение.
- Альтернативы: email/логин занят; пароль не проходит правила; ошибка сети.

### Задачи
**Frontend (Android)**
- Экран регистрации + валидации полей.
- Интеграция с API регистрации.
- UI состояния: loading/error/success.

**Backend (Spring Boot)**
- Endpoint регистрации (DTO + валидация).
- Сохранение пользователя (JPA).
- Обработка конфликтов (уникальность логина/email).

---

## 2) Авторизация

### User Story
Как сотрудник, я хочу входить в систему, чтобы пользоваться расписанием и встречами.

### Use Cases
**UC-2.1 Вход**
- Предусловия: есть аккаунт.
- Основной сценарий: открыть вход → ввести логин+пароль → получить токен/сессию → перейти в приложение.
- Альтернативы: неверные данные; пользователь заблокирован/не найден; ошибка сети.

**UC-2.2 Выход**
- Основной сценарий: нажать “Выйти” → токен удалён локально → переход на экран входа.

### Задачи
**Frontend**
- Экран входа.
- Хранение токена (например, EncryptedSharedPreferences).
- Logout: очистка сессии.

**Backend**
- Endpoint login (выдача JWT/сессии — выбрать подход).
- Middleware/фильтр авторизации для защищённых эндпоинтов.

---

## 3) Настройка личного профиля

### User Story
Как сотрудник, я хочу настраивать профиль, чтобы коллеги понимали, кто я и как со мной связаться.

### Use Cases
**UC-3.1 Просмотр профиля**
- Предусловия: авторизован.
- Основной сценарий: открыть профиль → получить данные → показать.

**UC-3.2 Редактирование профиля**
- Основной сценарий: изменить поля → сохранить → обновить данные.
- Альтернативы: некорректные значения; ошибка сети.

### Задачи
**Frontend**
- Экран профиля (view/edit режимы).
- Интеграция GET/PUT профиля.
- Локальное обновление данных после сохранения.

**Backend**
- Endpoints: получить текущего пользователя, обновить профиль.
- JPA сущность/репозиторий, DTO маппинг.
- Валидация входных данных.

---

## 4) Создание встречи + приглашения

### User Story
Как сотрудник, я хочу создавать встречу и приглашать коллег, чтобы договориться о совместном времени.

### Use Cases
**UC-4.1 Создать встречу (слоты строго по часам)**
- Предусловия: авторизован.
- Основной сценарий: выбрать дату → выбрать слот (например 9:00–10:00) → название/описание (опционально) → выбрать участников → создать.
- Ограничения: время только “целый час”; конец = начало + N часов (если разрешите длительность >1ч).
- Альтернативы: слот занят (конфликт); приглашённые не найдены; ошибка сети.

**UC-4.2 Пригласить сотрудников**
- Основной сценарий: выбрать пользователей → отправить приглашения вместе со встречей.
- Альтернативы: часть пользователей недоступна/не существует.

### Задачи
**Frontend**
- Экран создания встречи:
  - выбор даты
  - выбор слота “по часам” (UI без минут)
  - выбор участников (поиск/список)
- Валидация: запрет минут, подсветка конфликтов по ответу сервера.
- Интеграция API создания встречи.

**Backend**
- Модель встреч/приглашений (Meeting, Invitation/Attendee).
- Endpoint создания встречи (создатель + список приглашённых).
- Проверка слотов: начало строго HH:00; конец кратен часу; конфликт по занятости.
- Транзакционное создание встречи + приглашений.

---

## 5) Просмотр активных приглашений + ответ (принял/не принял)

### User Story
Как приглашённый сотрудник, я хочу видеть активные приглашения и отвечать “принял/не принял”, чтобы организатор понимал мой статус.

### Use Cases
**UC-5.1 Просмотр списка активных приглашений**
- Предусловия: авторизован.
- Основной сценарий: открыть “Приглашения” → получить список со статусом PENDING → показать.
- Альтернативы: пустой список.

**UC-5.2 Ответить на приглашение (бинарно)**
- Основной сценарий: открыть приглашение → выбрать “принял” или “не принял” → отправить → обновить статус.
- Альтернативы: приглашение истекло/отменено; слот конфликтует с уже принятой встречей (если вы это запрещаете).

### Задачи1
**Frontend**
- Экран “Приглашения”: список + карточка встречи.
- Действия: Accept / Decline.
- UI обновление статуса без перезахода.

**Backend**
- Endpoint списка приглашений пользователя (фильтр по PENDING).
- Endpoint ответа на приглашение (PATCH/POST).
- Бизнес-правило: что делать при конфликте принятого слота (заранее определить — запрет/разрешить).

---

## 6) Просмотр собственного расписания (день/неделя/месяц)

### User Story
Как сотрудник, я хочу видеть своё расписание встреч по дням/неделям/месяцам, чтобы планировать работу.

### Use Cases
**UC-6.1 Расписание на день**
- Основной сценарий: выбрать дату → получить встречи → показать по часам.

**UC-6.2 Расписание на неделю**
- Основной сценарий: выбрать неделю → получить встречи диапазоном → показать по дням.

**UC-6.3 Расписание на месяц**
- Основной сценарий: выбрать месяц → получить встречи диапазоном → показать маркеры занятости/список по дням.

### Задачи
**Frontend**
- 3 представления (Day/Week/Month) с навигацией по датам.
- Отрисовка слотов строго по часам (особенно Day view).
- Интеграция API “расписание в диапазоне дат”.
